Identified shortcomings in the `daml_reason` tool flow and their fixes.
Ordered by execution sequence: structural cleanup first, then core content fix, then cheap wins, then LLM gating cluster.

| # | Problem | Affected Files | Fix |
|---|---|---|---|
| 1 | **Dual `_ensure_semantic_search()`** â€” `DamlReasonTool` and `SafetyChecker` each have identical initialisation logic. The tool injects its instance post-init but SafetyChecker's own guard can silently create a second ChromaDB connection if called before injection completes. | `daml_reason_tool.py`, `safety_checker.py` | Remove `SafetyChecker._ensure_semantic_search()` entirely. Make the injected search a required dependency; fail loudly if it is `None` when `check_pattern_safety` is called rather than silently creating a second connection. |
| 2 | **O(nĂ—k) linear scan in `search_similar_files`** â€” after ChromaDB returns top-k result IDs, the code iterates all `raw_resources` (potentially 10,000+) for each result to match by name or `file_path`. Beyond being wasteful, it is a correctness risk: any formatting difference between what was stored in ChromaDB metadata at index time and what is in `_raw_resources` at query time causes a silent match failure and the result returns with no enrichment. | `semantic_search.py` | Build a `file_path â†’ resource` lookup dict once when `_raw_resources` is populated. O(1) lookup per result, and the match is deterministic. |
| 3 | **Redundant second ChromaDB query in Case 2c** â€” when code fails validation and `safety_result.similar_files` is empty, `DamlReasonTool.execute()` runs a second `search_similar_files(code=daml_code)` query identical to the one SafetyChecker already ran. | `daml_reason_tool.py` | Remove the fallback query branch in `execute()`. `SafetyCheckResult` already carries `similar_files`; always use those. If they are empty, return an empty pattern list rather than re-querying. |
| 4 | **LLM receives empty content** â€” after ChromaDB returns top-k matches, `_raw_resources` has been stripped of file content to save RAM. `file.get("content", "")` always returns `""`. The LLM reasons over file names and 200-char descriptions only, with no actual code. | `semantic_search.py`, `safety_checker.py`, `daml_reason_tool.py` | After top-k is returned by ChromaDB, re-read each file from disk: `(CANONICAL_DOCS_PATH / source_repo / file_path).read_text()`. Both `source_repo` and `file_path` survive the stripping step and are available in every result. |
| 5 | **Module name regex drops dotted hierarchy** â€” `_extract_module_name` uses `r'^\s*module\s+(\w+)\s+where'` which captures only the first identifier. `Finance.Instruments.Bond` is passed as `Finance`. | `daml_reason_tool.py` | Change `\w+` to `[\w.]+` in the regex pattern. |
| 6 | **No similarity score threshold** â€” `search_similar_files` always returns top-k regardless of actual relevance. A completely unrelated query still returns 5 results with near-zero similarity and the LLM reasons over meaningless comparisons. | `semantic_search.py` | Filter results below `MIN_SIMILARITY_THRESHOLD` (default `0.3`, configurable via env var) before returning. An empty list naturally skips the LLM call via existing `if similar_files:` guards, returning an honest "no relevant patterns found" rather than a hallucinated analysis. |
| 7 | **Silent permissive fallback on LLM failure** â€” missing `ANTHROPIC_API_KEY` or any exception in `_check_safety_with_llm` returns `{"is_safe": True, "confidence": 0.5}`, passing all code through as safe. A dev-era fallback that should not exist in production. | `safety_checker.py` | Raise `RuntimeError` on missing key and re-raise on LLM exception. Catch in `DamlReasonTool.execute()` and return a clean `action: "delegate"` result with message `"Service temporarily unavailable â€” analysis requires LLM connectivity"`. Stack trace stays in server logs only. |
| 8 | **`ENABLE_LLM_ENRICHMENT` ignored in Case 2** â€” in the no-code path (Case 1) the LLM is correctly gated by `ENABLE_LLM_ENRICHMENT`. In the code-analysis path (Case 2) `_check_safety_with_llm` and `AuthorizationValidator` both call the LLM regardless of that flag. Inconsistent with the pricing model which charges 0.15 CC based on that env var. | `safety_checker.py`, `authorization_validator.py` | Gate all LLM calls in Case 2 on `get_env_bool("ENABLE_LLM_ENRICHMENT", False)` the same way Case 1 does. When disabled, return a deterministic non-LLM result with moderate confidence. |
| 9 | **Compilation context path is dead code** â€” `compilation_result` is hardcoded to `None` in `check_pattern_safety` and never reassigned even when the client sends `compilation_context`. A TODO comment acknowledges this. Consequently `TypeSafetyVerifier.verify_type_safety()` is never called. | `safety_checker.py` | Convert the incoming `compilation_context` dict to a proper `CompilationResult` object at the top of `check_pattern_safety` and assign it to `compilation_result`. |
